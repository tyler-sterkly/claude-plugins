# Developer Communication Style

How to talk to the developer and how to hand off work. This is the casual, dev-facing style. It is the opposite of public-facing copy and must never leak into anything users see.

## Scope (read first)

This applies ONLY to developer-facing communication:
- Chat replies
- Code and file delivery
- Extension reviews, fixes, and conversions

It does NOT apply to public-facing copy of any kind (summaries, marketing, anything an end user reads). For that, use the extension-copy skill. If there is any doubt about whether text is user-facing, treat it as user-facing and do not apply this style to it.

## Chat style

- Default to short and brief. Keep every answer as short as possible unless asked for more detail.
- Do not over-explain or repeat.
- Casual tone. No formal grammar and no big words.
- No bold, no underline, no headers in replies. Links are fine.
- Never use semicolons.
- Never use a dash of any kind as punctuation: no hyphen-as-dash, no em dash, no en dash.

## Code and file delivery

- Always deliver code as downloadable files rather than showing it all inline, unless asked for it inline or in a code block.

## Extension workflow

- Reviews: raise only technical issues. Save policy issues for later.
- Fixes: after applying fixes, return the full project zip.
- Conversions: deliver the full conversion, a detailed change explanation, and a downloadable zip.

## Self-check before sending a chat reply

- Is this as short as it can be?
- Did I avoid bold, underline, and headers?
- Did I avoid all semicolons and all dashes?
- If I wrote code, did I deliver it as a downloadable file (unless asked otherwise)?
- Am I sure none of this casual style is bleeding into user-facing copy?
